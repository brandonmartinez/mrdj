# Linus Frontend Review — 2026-06-24

## Overall health

The frontend is in decent shape structurally: TypeScript is strict, the Vite build passes, most async screens have loading/empty states, and React's default escaping keeps user-supplied song/org/member text out of obvious XSS paths. The main risks are correctness and resilience rather than style: a root render error can still blank the app, several management mutations mis-handle successful empty responses, auth expiry is swallowed as empty data, and queue/area state can desync under failures or rapid changes.

## Findings

| SEV | Title | Location | Problem | Recommended fix |
| --- | --- | --- | --- | --- |
| High | No app-level error boundary for blank-page failures | `web/src/main.tsx:9` | `createRoot(...).render(...)` mounts the entire router without an error boundary. Any runtime render error in a route/component, or a bad top-level import that reaches render, takes down the whole app with a blank page class of failure. | Add a root ErrorBoundary around `App` with a friendly recovery UI and logging; consider route-level boundaries for org console vs guest jukebox so one surface does not take down the other. |
| High | Successful DELETE/204 responses throw client errors | `web/src/api.ts:131`; affected callers `web/src/pages/EventManage.tsx:72`, `web/src/pages/Members.tsx:60`, `web/src/pages/Pricing.tsx:76` | `apiFetch` always calls `res.json()`. The org DELETE endpoints return 204/no body, so a successful delete resolves to a JSON parse error and the UI reports failure/stays stale even though the server already mutated state. | In `apiFetch`, return `undefined` for `res.status === 204` or empty bodies before parsing JSON. Keep callers typed as `void`. |
| Medium | 401/expired sessions are swallowed as empty org data | `web/src/api.ts:119`; examples `web/src/pages/EventsList.tsx:33`, `web/src/pages/Members.tsx:29`, `web/src/pages/Pricing.tsx:36`, `web/src/pages/Earnings.tsx:26` | Protected pages catch API failures and substitute `[]`/`null`. If the session expires after `RequireAuth` has passed, users see empty dashboards/tables instead of being sent to login, and the app keeps stale `isAuthed` state. | Centralize 401 handling in `apiFetch`/session context: clear session and redirect or expose an auth-expired state. Do not convert auth errors into empty successful data. |
| Medium | Guest jukebox can spin forever when queue load fails | `web/src/pages/GuestJukebox.tsx:88`; `web/src/hooks/useQueueStream.ts:38` | Initial `/me` and bundle loads can succeed while the queue endpoint/SSE fetch fails. `useQueueStream` silently ignores queue errors, and `loading` only becomes false in `handleQueueUpdate`, leaving an indefinite loading screen for event-not-found, 401, or queue outages. | Track queue initial-load error separately; after the first queue fetch fails, clear loading and show a retryable event/queue error instead of swallowing it. |
| Medium | DJ console area selector does not affect queue or admin actions | `web/src/pages/DJConsole.tsx:52`; `web/src/pages/DJConsole.tsx:40`; `web/src/pages/DJConsole.tsx:80` | `areaId` updates from the selector, but `useQueueStream(eventSlug, setQueueView)` omits it, and `AdminConsole` only receives `eventSlug`. The selected area is therefore UI-only; admins can believe they are managing one area while seeing/mutating the event-level/default queue. | Pass `areaId` through queue stream and admin mutation APIs, or remove/disable the selector until per-area console routing is actually supported. |
| Medium | Stale queue responses can overwrite newer area/slug state | `web/src/hooks/useQueueStream.ts:30` | Queue fetches are not abortable or sequence-checked. When slug/area changes, an older in-flight `api.queue(slug, areaId)` can resolve after the new effect starts and still call `onUpdate`, briefly or permanently showing the wrong queue/balance. | Use an `AbortController` or monotonically increasing request id inside the effect; ignore responses from cleaned-up effects. |
| Medium | Empty search query still hits the API and can render unintended results | `web/src/pages/GuestJukebox.tsx:106`; `web/src/pages/GuestJukebox.tsx:319` | The search effect calls `api.search(debouncedQuery)` even when the query is empty. If the backend returns popular/all tracks for `q=`, those results render because the results section only checks `searchResults.length`, cluttering the guest view and adding avoidable load. | Skip search for blank/whitespace queries, clear `searchResults`, and only render result rows when `debouncedQuery.trim()` is non-empty. |
| Medium | Pricing UI allows zero-credit bundles and bad credit math | `web/src/pages/Pricing.tsx:152`; `web/src/pages/Pricing.tsx:157`; `web/src/components/ConfirmModal.tsx:345` | Admins can save a bundle with `credits=0` and `bonusCredits=0`; guest checkout then computes `price / totalCredits`, producing `Infinity`/invalid per-credit display and a nonsensical purchasable product if the server allows it. | Validate `credits + bonusCredits > 0` and `priceCents > 0` client-side before save; defensively guard the guest per-credit display. |
| Low | Dev auth controls are exposed in the normal UI | `web/src/components/Header.tsx:91`; `web/src/pages/Login.tsx:52` | The guest header always shows Guest/Admin role-switch buttons, and login always shows Dev sign-in. Even if the backend rejects these in production, guests see confusing controls; if the dev route is accidentally enabled, this becomes privilege escalation. | Gate dev controls behind an explicit dev feature flag/environment check and hide them from production bundles. |
| Low | Custom modal lacks focus trap and focus restore | `web/src/components/ConfirmModal.tsx:62`; `web/src/components/ConfirmModal.tsx:69` | The modal focuses a button and handles Escape, but tab focus can leave the dialog behind the overlay and focus is not restored to the invoking track action on close. This is an a11y regression for keyboard/screen-reader users during credit purchase/request flows. | Use the existing shadcn/Radix `Dialog` primitive or implement focus trapping, inert background, and focus restoration. |

## What's solid

- Strict TypeScript build passes: `npm run build -- --emptyOutDir=false` completed successfully.
- Queue rendering uses stable keys and React text escaping; no `dangerouslySetInnerHTML` usage was found.
- Guest queue has SSE plus fallback polling, and most org pages include reasonable skeleton/empty states.
- Money display consistently formats cents with fixed two-decimal dollar strings in dashboard/earnings/pricing.
