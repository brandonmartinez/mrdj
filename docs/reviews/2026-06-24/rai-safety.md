# RAI safety/privacy review — 2026-06-24

## Trust & safety posture

mrdj has a solid MVP safety baseline: privileged org/admin routes are explicitly guarded, Stripe card data is delegated to Stripe, webhook signatures are verified, React's default escaping avoids obvious XSS from displayed text, and `.gitignore` protects local env/secret files. The main trust blocker is that anonymous guests are not actually separated: every new unauthenticated browser is assigned the same seeded guest user, so wallet balance, paid-credit spend authority, queue attribution, and payment metadata can collapse across unrelated people. Privacy posture is otherwise incomplete for a marketplace: account emails/display names and payment/user ledgers are stored, but retention/export/deletion expectations are not documented in-repo.

## What's handled well

- Stripe card details are handled by Stripe Payment Element / PaymentIntents; the server stores Stripe ids and ledger metadata, not card numbers (`web/src/components/StripeCheckout.tsx`, `api/src/payments/purchase.ts`).
- Stripe webhooks verify signatures against the raw body and use idempotent processing (`api/src/http/server.ts:38-42`, `api/src/payments/webhooks.ts:27-84`).
- Public/admin/org route boundaries are explicit, with org membership guards on org-scoped management routes and production gates on dev/stub checkout endpoints (`api/src/http/routes.ts:93-138,160-224`).
- React renders user/provider/org strings as text rather than raw HTML; no `dangerouslySetInnerHTML` usage was found.
- `.gitignore` covers root and k8s secret env files, and tracked env files are examples/placeholders only.

## Secret scan result

No committed real secrets found in current tracked files or refined git-history scans. Local ignored `.env`/`k8s` env files are present and should be treated as sensitive operational material, but they are not tracked.
