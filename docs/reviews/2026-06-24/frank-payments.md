# Frank Payments Review — 2026-06-24

## Overall health

Payments/credits are in relatively good shape for an MVP: Stripe webhooks use raw-body signature verification, purchase amounts come from server-side org bundles, payment-intent and credit-grant idempotency guards are present, wallet debits are transactional with queue writes, and the wallet has a database non-negative CHECK. The highest-risk remaining gaps are around refund state consistency and reconciliation: the code can return credits without marking the payment refunded, webhook grants trust Stripe metadata instead of a server-owned purchase record, and mutable wallet balances are not regularly reconciled back to the append-only ledger.

## What's solid

- Stripe webhook route is mounted with `express.raw` before JSON parsing and verifies `stripe-signature` (`api/src/http/server.ts:40`, `api/src/payments/webhooks.ts:37`).
- Webhook idempotency records processed event IDs and also guards grants with unique `stripe_payment_intent_id` and credit idempotency keys (`api/src/payments/webhooks.ts:47`, `api/src/payments/webhooks.ts:119`).
- Purchases use server-side bundle price/credits and Stripe destination charges with integer cents and application fees (`api/src/payments/purchase.ts:32`, `api/src/payments/purchase.ts:58`).
- Paid queue requests read server pricing, run debit + queue insert + play-next slot update in one DB transaction, and use `wallets_balance_check` as a concurrent overspend backstop (`api/src/queue/index.ts:294`, `api/src/queue/index.ts:413`, `api/src/db/schema.ts:170`).
- Tests cover webhook replay, signature rejection, destination charge creation, refund basics, cross-org wallets, and concurrent spend caps (`api/src/__tests__/payments.test.ts`, `api/src/__tests__/money.test.ts`).
