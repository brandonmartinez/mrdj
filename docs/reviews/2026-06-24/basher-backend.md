# Basher backend review — 2026-06-24

## Overall health

The backend has a solid structure: route registration is explicit, tenant-scoped org/event/admin CRUD generally composes `resolveOrg()` + `requireMembership()`, async route failures are centrally caught, and the queue money path is transaction-oriented with Play Next row locks and idempotency recovery. The biggest risks I found are around production identity/session assumptions and lifecycle gaps introduced by multi-area/event creation: anonymous users are still collapsed onto one seeded guest user, OAuth state can be bypassed on direct callbacks, and newly created events do not receive the Play Next slot that the queue state machine requires.

## Findings

| Sev | Title | Location | What's wrong | Recommended fix |
| --- | --- | --- | --- | --- |
| **High** | All anonymous visitors share the same guest identity and wallet | `api/src/http/middleware.ts:27` | `initSession` assigns every session without `userId` to the same seeded `SEED_IDS.guestUser`. In production this means unrelated guests share requester identity, credit balance, credit-spend authority, and queue attribution across all browsers; rate limiting is per-session/IP, but authorization/accounting is not. | On first anonymous visit, create or load a unique guest `users` + `guest_sessions` row and store that user id in the session. Keep the seeded guest only for tests/dev fixtures, gated by `cfg.isDev` or test setup. |
| **High** | Created events lack a Play Next slot | `api/src/event/handlers.ts:131` | `createEventHandler` inserts the mandatory default `areas` row but never inserts the corresponding `play_next_slot`. `buildQueueView` then reports Play Next as available when no slot exists, while `createRequestHandler` treats `!slot` as unavailable (`api/src/queue/index.ts:332`), so Play Next is broken for newly created events. Seeded events and areas created later do get slots, which hides this in most tests. | In the same transaction that creates the default area, insert `play_next_slot` for that area. Add a regression test that creates an event and immediately purchases Play Next in its default area. |
| **Medium** | OAuth callback accepts requests without issued state | `api/src/auth/index.ts:55` | The callback only validates state when `req.session.oauthState` exists; otherwise it skips validation “for tests.” A direct browser hit to `/api/auth/google/callback?code=...` can establish a session without a prior login-start state in that browser, enabling login CSRF/session fixation style attacks. | Require both stored state and matching query state in all non-test runtime paths. Move test bypass behind an explicit test-only provider/helper instead of production handler logic. |
| **Medium** | Queue idempotency keys are global and unscoped | `api/src/queue/index.ts:321` | A retry lookup by `idempotencyKey` does not constrain `userId`, organization, event, area, tier, or route. If a client reuses a key accidentally (or another user obtains/guesses it), the handler returns the original queue item/result in the current user's response, potentially leaking another request's `queueItem` and confusing wallet/queue state. | Scope idempotency keys at lookup and uniqueness time: include `userId` + `organizationId` + operation namespace, or add a stable operation table keyed by `(user_id, organization_id, idempotency_key)`. On mismatch, return 409 instead of replaying another principal's result. |
| **Medium** | Malformed request fields can bubble into DB 500s | `api/src/queue/index.ts:256` | Queue submission checks only truthiness and tier membership. Non-string or malformed UUID values for `trackId` / `areaId` / `idempotencyKey` can reach Drizzle/Postgres and surface as `22P02`/500 instead of a 400; similar casts appear in admin/org/event handlers. | Add shared request validation for UUIDs, string length, and object shape before DB calls; translate Postgres `22P02` to a validation error only as a fallback. |
| **Low** | Last-owner protection is race-prone | `api/src/org/handlers.ts:224` | `updateMemberHandler` and `removeMemberHandler` check `ownerCount()` before updating/deleting, but the check and mutation are not in one locked transaction. Concurrent owner removals/demotions can both observe two owners and commit, leaving an organization with no owner. | Wrap last-owner checks and membership updates/deletes in a transaction and lock owner membership rows (`FOR UPDATE`), or enforce via a DB constraint/trigger. |

## What's solid

The queue write path is much stronger than the usual MVP baseline: debit, ledger insert, queue insert, and Play Next state updates happen in one transaction; Play Next purchases lock the per-area slot with `FOR UPDATE`; overspend/idempotency races have explicit Postgres-code recovery; and realtime payloads intentionally carry invalidation-only data so SSE does not leak per-user balances.
